# 日本語 UI ライティングガイド

日本語でUIテキストを書く際に、一貫性・簡潔性・わかりやすさを確保するためのガイドライン。
表記揺れの防止、冗長表現の排除、適切なUIテキストパターンの適用を通じて、
ユーザーの読解負荷を軽減する。

## テーマ一覧と行動指針

### 1. 基本的な考え方

**核心**: 冗長な日本語を避ける、敬語を使いすぎない、用語を統一する

UIテキストを書くとき → 「することができます」を「できます」に、「ご確認いただけますようお願いいたします」を「確認してください」に簡潔化する。同じ概念には同じ言葉を使い、社内用語や専門用語を避ける。

### 2. 用字用語

**核心**: 漢字/ひらがなの使い分け、カタカナ長音ルール、記号の全角/半角統一

UIテキストを書くとき → 読みやすさのために漢字をひらがなに開くか判断する。カタカナ語の長音（-er/-or → あり、-y → なし）を統一する。括弧の種類を用途別に使い分ける。

### 3. UIテキスト

**核心**: ボタンは動詞終止形、タイトルは名詞、ステータスは段階別パターン

UIテキストを書くとき → ボタンラベルには「追加」「削除」のように動詞の終止形を使う。画面タイトルには名詞を使う。ステータスラベルは「○○前」「○○中」「○○済み」の体系に従う。

### 4. エラーメッセージ

**核心**: 事象・原因・対処の3要素、スペース制限時は原因を優先

エラーメッセージを書くとき → 「何が起きたか」「なぜ起きたか」「どう解消するか」の3要素を含める。表示スペースが限られる場合は原因と対処を優先し、ユーザーが次のアクションを取れる状態にする。

## ライティングチェックリスト

UIテキストを書く・レビューする際に以下を確認すること：

**簡潔さ**
- [ ] 冗長な表現が残っていないか（「〜することができます」→「〜できます」）
- [ ] 敬語が過剰でないか（情報伝達を優先しているか）
- [ ] 社内用語や専門用語がユーザー向けの言葉に置き換えられているか

**表記の統一**
- [ ] 漢字/ひらがなの表記が統一されているか
- [ ] カタカナ語の長音が統一されているか（-er → 長音あり、-y → なし）
- [ ] 数字は算用数字で統一されているか（固有名詞・慣用句を除く）
- [ ] 記号の全角/半角が統一されているか

**UIテキストパターン**
- [ ] ボタンラベルは動詞の終止形になっているか（「追加」「削除」）
- [ ] 画面タイトル・項目名は名詞になっているか
- [ ] テキストリンクは移動先を想起できる具体的な表現か（「こちら」を避ける）
- [ ] ステータスラベルが段階別パターンに従っているか

**エラーメッセージ**
- [ ] 事象・原因・対処の3要素を含んでいるか
- [ ] ユーザーが次のアクションを取れる具体的な内容か
- [ ] スペース制限時に原因が省略されていないか
