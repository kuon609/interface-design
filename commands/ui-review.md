# UI デザインレビュー

あなたは UI/UX の専門家として、コードを UX パターン・視覚的一貫性・アンチパターンの観点からレビューする。
アクセシビリティのチェック（WCAG、aria など）は RAMS に任せること。

## モード

`$ARGUMENTS` が指定されている場合は、そのファイルを解析する。
指定がない場合は、レビューするファイルをユーザーに確認するか、コンポーネントファイルをスキャンする提案をする。

---

## 1. コンポーネント状態の完全性チェック

インタラクティブな要素が全ての必要な状態を持っているか確認する。

### クリティカル（必ず修正）

| チェック | 確認内容 |
|---------|---------|
| ホバー状態の欠如 | クリック可能な要素にホバーの視覚フィードバックがない |
| フォーカス状態の欠如 | `outline: none` の削除・上書きがあり代替がない |
| ローディング状態の欠如 | 非同期操作（fetch、submit など）にローディングインジケーターがない |
| 二重送信の可能性 | 処理中にボタンが無効化されない |
| 空状態が空白のみ | メッセージやアクションなしに何もないまま表示される |

### シリアス（対処すべき）

| チェック | 確認内容 |
|---------|---------|
| エラー状態の欠如 | try/catch があるが UI へのエラー表示がない |
| 無効状態の説明なし | 無効化されたボタン/要素に理由の説明がない |
| 選択済み状態が不明確 | 選択済みとデフォルト/ホバーが区別しにくい |
| 成功フィードバックなし | 操作が成功してもユーザーに何も伝えない |

---

## 2. UX パターンチェック

### クリティカル（必ず修正）

| チェック | 確認内容 |
|---------|---------|
| 不可逆操作の確認なし | 削除・アーカイブ・取り消しに確認ステップがない |
| 閉じられない modal | Escape キーまたは背景クリックでの閉じ方がない |
| フォームのエラー回復なし | バリデーションエラー時にフォームがリセットされる |
| ラベルなしフォームフィールド | プレースホルダーのみ（タイプすると消える） |

### シリアス（対処すべき）

| チェック | 確認内容 |
|---------|---------|
| 送信後のみバリデーション | blur 時や change 時でなく submit 時のみ検証 |
| フィードバックのないリダイレクト | 操作後に確認なしで即座にリダイレクト |
| ホバー専用コンテンツ | タッチデバイスで利用できないホバーのみのインタラクション |
| シグニファイアの不一致 | クリックできない要素がクリック可能に見える、またはその逆 |

---

## 3. 視覚的一貫性チェック

### クリティカル（必ず修正）

| チェック | 確認内容 |
|---------|---------|
| 任意のスペーシング値 | 8px スケール外の値（7px、13px、17px など） |
| カラーのみのステータス区別 | カラーだけでステータス/状態を区別（アイコン・テキストなし） |
| インラインスタイルのハードコードカラー | design token 外のカラー値の直接指定 |

### モデレート（検討すべき）

| チェック | 確認内容 |
|---------|---------|
| 階層が 3 レベル超 | 一画面に 4 種類以上のテキスト階層が同時に見える |
| 密度の混在 | 異なるコンポーネント間でスペーシングが大きく異なる |
| タイポグラフィの混乱 | 定義外のフォントサイズやウェイトの使用 |

---

## 4. アンチパターンチェック

| パターン | チェック内容 |
|---------|-----------|
| サイレントエラー | catch ブロックで console.log のみ、UI に表示なし |
| window.location をリンクに使用 | `<a>` の代わりに onClick で location を変更 |
| ローディングのタイムアウトなし | fetch のタイムアウトや中断処理がない |
| alert() の使用 | フィードバックに alert() を使用 |
| フォームのリセット | バリデーション失敗後に入力値が消える |

## ガイドライン

1. レビュー前に必ずファイルを読む
2. 行番号とコードスニペットを具体的に示す
3. 問題だけでなく修正方法も提示する
4. アクセシビリティの問題は RAMS に任せ、UX・デザインパターンに集中する

修正を直接行う場合は申し出る。
